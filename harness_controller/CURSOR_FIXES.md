# Cursor Blocking Issues — Fixed

This document tracks the issues Cursor struggled with and how they were fixed.

---

## Issue 1: RetrievalChunkSet Schema Unclear

### Problem
Cursor couldn't figure out that `similar_case` chunks must include a `task_shape` object.

**Error observed:**
```
Harness rejected the RetrievalChunkSet because the similar_case chunk (c5) 
must include a task_shape object.
```

### Root Cause
The skill documentation provided only a vague schema. It didn't explicitly show:
- What `task_shape` is
- When it's required (for `similar_case` chunks)
- What fields it contains
- Why it's needed

### Fix
Created `05-artifact-schemas-detailed.mdc` with:

✅ **Complete `similar_case` schema with `task_shape`:**
```json
{
  "chunk_id": "c_similar",
  "chunk_type": "similar_case",
  "task_shape": {
    "task_type": "feature|refactor|bugfix|etc",
    "complexity": "low|medium|high",
    "involved_layers": ["presentation", "api", "database"],
    "risk_level": "low|medium|high"
  }
}
```

✅ **Coverage rules section** explicitly stating:
- "Similar_case chunk required IF complexity is medium or high"
- "Similar_case chunks MUST include task_shape object"

✅ **Three complete examples** (low, medium, high complexity) showing:
- When `similar_case` is required
- How to structure `task_shape`
- What fields are mandatory

## Issue 2: ChunkQualityReport Fields Unknown

### Problem
Cursor didn't know about required fields in `ChunkQualityReport`:
- `errors` array
- `warnings` array
- Boolean flags: `has_core_task`, `has_constraint`, `has_risk`, `has_pattern`, `has_similar_case`

**Error observed:**
```
The harness is asking for more fields in ChunkQualityReport: 
it requires errors, warnings, and boolean flags like has_core_task, 
has_constraint, has_risk, has_pattern, and has_similar_case.
```

### Root Cause
The original schema in `00-harness-control-plane.mdc` was incomplete. It showed a minimal structure but didn't document:
- All required fields
- Boolean validation flags
- Error/warning fields
- Why these fields matter

### Fix
Created `05-artifact-schemas-detailed.mdc` with:

✅ **Complete `ChunkQualityReport` schema with ALL fields:**
```json
{
  "validation_summary": "...",
  "chunk_assessments": [...],
  "overall_coverage_adequate": true|false,
  "gaps_identified": [],
  "errors": [],
  "warnings": [],
  "has_core_task": true|false,
  "has_constraint": true|false,
  "has_risk": true|false,
  "has_pattern": true|false,
  "has_similar_case": true|false
}
```

✅ **Field reference table** documenting each field

✅ **Validation rules section** explicitly stating:
- "has_core_task must be true — always required"
- "has_constraint must be true IF RequirementIntent.hard_constraints is non-empty"
- "has_similar_case must be true IF RequirementIntent.complexity is medium or high"

✅ **Complete working example** with all fields populated

✅ **Common mistakes section** showing what NOT to do:
- Missing boolean flags
- Using strings instead of booleans
- Wrong values for flags

## Issue 3: MCP Payload Schema Mismatch

### Problem
Cursor submitted `chunkType` as strings (e.g., `"core_task"`), but the local MCP tool expected PascalCase (e.g., `"CoreTask"`).

**Error observed:**
```
The MCP call still fails because the local retrieve_memory_by_chunks tool 
can't invoke with the request structure harness produced (it returned 
"An error occurred invoking 'retrieve_memory_by_chunks'"). This suggests 
the harness is generating an MCP payload that doesn't match the local MCP 
tool's expected schema (e.g., chunkType is an integer there, but we're 
currently passing string values).
```

### Root Cause
The skill documentation didn't clarify:
- The agent uses `chunk_type: "core_task"` (snake_case, lowercase)
- Harness converts this to `chunkType: "CoreTask"` (PascalCase) for MCP
- The MCP tool expects PascalCase enum values
- The conversion is automatic and correct

### Fix
Created `05-artifact-schemas-detailed.mdc` with:

✅ **Explicit section: "MCP Request Structure (What Harness Provides)"** stating:
- "Use `payload.request` exactly as provided. Do not modify it."
- "The `chunkType` in the MCP request uses PascalCase (`CoreTask`, `SimilarCase`), NOT snake_case (`core_task`)."
- "Harness automatically converts your `chunk_type: "core_task"` from RetrievalChunkSet to `chunkType: "CoreTask"` for MCP."

✅ **Side-by-side comparison:**
```
Agent schema (RetrievalChunkSet):
  "chunk_type": "core_task"  ← snake_case

MCP request (generated by harness):
  "chunkType": "CoreTask"    ← PascalCase (automatic conversion)
```

✅ **MCP response section** explicitly stating:
- "Submit this response exactly as received. Do not modify it."
- "Do NOT modify, transform, or wrap it."

## How Cursor Should Use These Fixes

When generating artifacts:

1. **Always check the detailed schema file first:**
   - `harness/agent-rules/05-artifact-schemas-detailed.mdc`
   - It has complete, correct schemas with examples

2. **For RetrievalChunkSet:**
   - Read the "Coverage Rules" section
   - Check the three complete examples (low, medium, high complexity)
   - Verify `similar_case` chunks have `task_shape` if complexity is medium/high

3. **For ChunkQualityReport:**
   - Copy the exact schema from the detailed schema file
   - Include ALL boolean flags: `has_core_task`, `has_constraint`, `has_risk`, `has_pattern`, `has_similar_case`
   - Verify flags match the actual chunks

4. **For MCP calls:**
   - Understand that harness converts `chunk_type` to `chunkType` automatically
   - Submit raw, unmodified MCP responses
   - Don't try to "fix" the response
